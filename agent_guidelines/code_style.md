# 代码规范

## C++ 风格

- 遵守仓库现有 `.clang-format`：
  - Google 风格
  - 4 空格缩进
  - 100 列限制
- 优先使用清晰的小函数和 adapter，避免把型号差异散落到业务逻辑中。
- 不写“解释下一行代码”的低价值注释；只在以下情况写注释：
  - 坐标系、单位、角度方向容易误用。
  - 安全限制或禁止项必须被未来维护者看到。
  - 外部 SDK / DDS topic 行为不直观。
- 不引入全局 mutable 状态，除非现有模块模式已经如此且有明确初始化/关闭顺序。
- 新增 magic number 必须集中定义，并说明单位。

## CMake / 构建风格

- 机器人型号差异必须通过明确分支隔离：
  - `ROBOT_MODEL_K1`
  - `ROBOT_MODEL_T1`
  - `ROBOT_MODEL_G1`
- G1 SDK 依赖必须封装在 G1 专用 CMake 逻辑中，不得污染 K1/T1。
- K1/T1 现有 build option 必须保持兼容：
  - `--k1z`
  - `--k1bc`
  - `--t1`
  - `--t1_robocup_version`
- 新增 G1 构建应使用：
  - `ROBOT_MODEL=g1`
  - `build-g1`
  - `install.bash --g1`

## 目录和命名

- G1 专用适配层优先命名为 `g1_*_adapter` 或 `motion_connector_g1`。
- 通用 pub/sub 类型不得强行塞入 G1 31DOF 特例；确实需要时新增 G1 专用类型。
- 新增文件应放在与现有模块对应的目录下，避免建立无关的大型新框架。

## 改动范围

- 不做无关重构。
- 不批量格式化未修改文件。
- 不为了 G1 改掉 K1/T1 已稳定接口，除非同时保留兼容层。
- 若发现旧代码有风险但不属于当前任务，记录 TODO 或在说明中指出，不顺手大改。
